# Firmware migration: 1.0.36 to 1.0.37

## Release status

The original three-run OTA gate passed on 2026-07-16. A later visual check found
a Mini GIF loop regression and a stale factory Mini label. The corrected
candidate from commit `77a1726` has passed one signed-app `1.0.36 -> 1.0.37`
migration. Visual confirmation and the remaining repeated runs are recorded in
`docs/control-center-hardware-test-evidence.md`. Merge, tag, and release remain
separate guarded actions.

## Why this migration needs a compatibility mode

Firmware `1.0.36` can accept an OTA TCP connection but stop draining upload
bytes. A normal macOS sender then buffers roughly `130 KB` before ending with a
broken pipe. The target `1.0.37` binary is not validated in this failure mode
because it never arrives completely.

The `1.0.36` RAW handler also leaves the ESP8266 `Update` object active after a
timeout. A second upload in the same boot can then fail with the misleading
message `Update failed: No Error`. A power restart clears that state.

## Customer-safe migration behavior

- Control Center downloads and validates the release artifact before upload.
- It pins the selected device and pauses the display stream.
- The Companion uses the paced RAW transport specified in
  `docs/firmware-ota-contract.md`.
- If the upload response is lost but `1.0.37` returns, the update is treated as
  successful.
- If `1.0.36` returns after an interrupted upload, Control Center stops. It must
  not attempt multipart or another upload in the same boot.
- Recovery does not require a data USB connection: disconnect VibeTV power for
  10 seconds, reconnect it, wait for the picture, and retry once.
- ACK timeouts, socket timeouts, lost final responses, and the known
  `Update failed: No Error` response are treated as potentially partial
  uploads. The Companion transports a typed `power_cycle` retry policy to
  Control Center instead of relying on the child-process error text.
- Control Center does not show `Try again` while that `power_cycle` policy is
  active. It shows the power-disconnect recovery and support-report action.
- A retry after repairing an expired pairing token goes through the same
  interrupted-upload recovery classification.

Automated coverage includes ACK timeout after upload start, the stale
`No Error` state, a lost final response, the API error mapping, and the absence
of the same-boot retry button in the customer UI.

## Changes included in the 1.0.37 receiver

- non-blocking partial socket reads instead of `readBytes()` for full 1 KB
  blocks;
- exact RAW body-size validation;
- unrelated TCP and UDP connections closed before flashing;
- renderer and filesystem resources released before update mode;
- one cleanup path for every ESP8266 updater failure;
- controlled restart after an OTA failure that entered update mode.

These changes protect `1.0.37 -> later` updates. They cannot repair devices
while they are still executing `1.0.36`; the paced Control Center sender is the
bridge for that installed base.

## Preserved Mini theme compatibility

Firmware OTA preserves LittleFS. Existing `1.0.36` devices therefore keep the
factory Mini spec at `/themes/u/mini-cl-1-410a37.json`, including its two
hard-coded `left` labels. Firmware `1.0.37` recognizes only that immutable
legacy factory path and substitutes the live `{usageMode}` binding in memory.
It does not rewrite customer storage. This also covers untouched devices that
have the old factory file but no `/theme-active` pointer.

New filesystem images and provisioning use the current
`/themes/u/mini-cl-1-b3c3f7.json` spec. The Mini GIF itself is unchanged and
matches the current theme pack: `20870` bytes, SHA-256
`146563e7d7ce76589f50ea80977c1f8cb32a15468a4a9f3da4152c4de1b37230`.

The GIF loop reset now keeps decoder reset, background clear, and first-frame
decode inside the same display transaction. This restores the transaction
boundary used by `1.0.36` without reintroducing a global transaction around the
cooperative animation tick.
